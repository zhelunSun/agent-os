# Schemas

Data schemas for all Agent OS file types.

---

## 1. Finding (`findings.jsonl`)

Each line is a JSON object. File is append-friendly; updates rewrite the matching line in-place.

```json
{
  "id": "AF-YYYYMMDD-NNN",
  "date": "YYYY-MM-DD",
  "source": "string — e.g. daily-audit, code-review, manual",
  "repo": "string — repository or project identifier",
  "severity": "P0 | P1 | P2 | P3",
  "status": "open | accepted | fixed | verified | closed | deferred | false_positive",
  "title": "string — concise one-liner",
  "owner": "string — agent or human responsible",
  "evidence": ["string — file paths, URLs, references"],
  "next_action": "string — what needs to happen next",
  "verification": "string — how to confirm it's resolved",
  "tags": ["optional", "tags"],
  "updated_at": "YYYY-MM-DDTHH:MM:SS"
}
```

### Status Lifecycle

```
open → accepted → fixed → verified → closed
  │                                ↑
  └──→ deferred ──→ open          │
  └──→ false_positive → closed ←──┘
```

### ID Convention

- Format: `AF-YYYYMMDD-NNN` where NNN is a zero-padded sequence number
- Example: `AF-20260601-001`
- IDs are stable — never reused or reassigned

## 2. Context Pack (`context-packs/YYYY-MM-DD-<label>.md`)

Markdown file with structured sections:

```markdown
# Context Pack: <label>

**Date**: YYYY-MM-DD
**Task**: <what this pack is for>
**Generated by**: <agent name>

## Current Goal

<1-2 sentences describing the immediate objective>

## Must-Read Files

- `<path>` — <why it matters>

## Open Blockers

- <list of open P0/P1 findings relevant to this task>

## Active Repos / Worktrees

- <repo> — <status: clean/dirty, branch>

## Forbidden Actions

- <things the agent must NOT do>

## Expected Output

- <what success looks like>
```

## 3. Mission Control (`mission-control/YYYY-MM-DD.md`)

One-page project health view in Markdown:

```markdown
# Mission Control — YYYY-MM-DD

## Overall Health: <score>/100

## Active Projects

| Project | Status | Last Update | Open Findings |
|---------|--------|-------------|---------------|
| <name>  | <ok/warn/crit> | YYYY-MM-DD | N |

## Open Findings (P0/P1)

| ID | Title | Severity | Owner | Age |
|----|-------|----------|-------|-----|
| AF-... | <title> | P0/P1 | <owner> | Nd |

## Stale Items

- <items that haven't been updated in >48h>

## Human Decisions Needed

### Decision 1: <title>
- **Recommendation**: <what to do>
- **Why now**: <urgency>
- **Options**: A) ... B) ...
- **Consequence of inaction**: <what happens if delayed>
- **Agent can continue**: <what doesn't block on this>

## Agent Activity Summary

- <brief summary of what agents did today>

## Metrics

| Metric | Value |
|--------|-------|
| Open P0 | N |
| Open P1 | N |
| Avg finding age | Nd |
| Findings closed today | N |
```

## 4. Flight Record (`flight-recorder/YYYY-MM-DD.jsonl`)

Each line is a JSON object representing one agent run:

```json
{
  "run_id": "FR-YYYYMMDD-HHMMSS",
  "timestamp": "YYYY-MM-DDTHH:MM:SS",
  "agent": "string — agent identifier",
  "automation": "string — optional automation name",
  "task": "string — what the agent was asked to do",
  "context_summary": "string — brief description of inputs consumed",
  "decisions": [
    "string — key decision made and why"
  ],
  "files_changed": ["string — paths"],
  "commands_run": ["string — commands executed"],
  "verification": "string — how results were verified",
  "failures": ["string — errors or anomalies"],
  "downstream_impact": "string — what this affects",
  "findings_created": ["AF-..."],
  "findings_updated": ["AF-..."]
}
```

## 5. Experience Log (`experience-log.jsonl`)

Classified learnings from agent runs:

```json
{
  "id": "EX-YYYYMMDD-NNN",
  "date": "YYYY-MM-DD",
  "source": "string — finding, incident, manual",
  "learning": "string — what was learned",
  "classification": "rule | test | gate | prompt | doc",
  "target": "string — where this should be applied",
  "status": "draft | approved | implemented | verified",
  "evidence": ["string — supporting references"]
}
```

### Classification Guide

| Class | Meaning | Example Target |
|-------|---------|----------------|
| `rule` | A behavioral constraint for agents | Agent rules file |
| `test` | Should become an automated test | Repo test suite |
| `gate` | A release/deployment check | CI pipeline |
| `prompt` | Update to an automation prompt | Automation config |
| `doc` | Documentation-only knowledge | Knowledge base |
